# SEO & AEO Optimization Skill

This skill ensures all pages and content on the ABRAM documentation site are optimized for both traditional search engines (Google, Bing) and AI-powered answer engines (ChatGPT, Perplexity, Google AI Overviews).

## Quick Reference: New Page Checklist

Before completing any new page, verify ALL of these are present:

- [ ] **Page is a Server Component** (no `"use client"` in `page.tsx`) OR uses the server-wrapper pattern
- [ ] **`metadata` export** with: title, description, keywords, canonical, openGraph, twitter
- [ ] **JSON-LD structured data** via `<script type="application/ld+json">`
- [ ] **Semantic HTML** (`<main>`, `<article>`, `<section>`, `<nav>` where appropriate)
- [ ] **Single `<h1>` per page**
- [ ] **Updated `docs.json`** (if it's a doc article)
- [ ] **Updated `public/llms.txt`** (if it's a doc article)
- [ ] **Rebuilt search index**: `node scripts/build-search-index.js`
- [ ] **Rebuilt LLM file**: `node scripts/build-llms-txt.js`
- [ ] **Build passes**: `npm run build`

## 1. Server Component Metadata Pattern

Next.js App Router requires metadata exports from **Server Components only**. If a page needs client-side interactivity, use the server-wrapper pattern:

### Pattern: Server Component Page (preferred)
```tsx
// src/app/my-page/page.tsx
import type { Metadata } from 'next';

export const metadata: Metadata = {
  title: 'Page Title',  // Will become "Page Title | ABRAM Docs" via root template
  description: 'Page description under 160 characters.',
  keywords: ['ABRAM', 'keyword1', 'keyword2'],
  alternates: {
    canonical: 'https://docs.abram.network/my-page',
  },
  openGraph: {
    title: 'Page Title — ABRAM',
    description: 'Page description for social shares.',
    type: 'website',
    url: 'https://docs.abram.network/my-page',
  },
  twitter: {
    card: 'summary_large_image',
    title: 'Page Title — ABRAM',
    description: 'Page description for Twitter/X.',
  },
};

export default function MyPage() {
  return (
    <>
      <script
        type="application/ld+json"
        dangerouslySetInnerHTML={{
          __html: JSON.stringify({
            '@context': 'https://schema.org',
            '@type': 'WebPage',
            name: 'Page Title',
            description: 'Page description.',
            url: 'https://docs.abram.network/my-page',
            isPartOf: { '@id': 'https://docs.abram.network/#website' },
            publisher: { '@id': 'https://abram.network/#organization' },
          }).replace(/</g, '\\u003c'),
        }}
      />
      <main>
        {/* Page content */}
      </main>
    </>
  );
}
```

### Pattern: Client Component with Server Wrapper
```tsx
// src/app/my-page/MyPageClient.tsx
"use client";
// ... all interactive logic here

// src/app/my-page/page.tsx (SERVER component — NO "use client")
import type { Metadata } from 'next';
import MyPageClient from './MyPageClient';

export const metadata: Metadata = { /* same pattern as above */ };

export default function MyPage() {
  return (
    <>
      <script type="application/ld+json" ... />
      <MyPageClient />
    </>
  );
}
```

## 2. Metadata Field Requirements

### Required for ALL pages:
| Field | Description | Example |
|-------|-------------|--------|
| `title` | Page title (root template appends " \| ABRAM Docs") | `'AI Brief Analyzer'` |
| `description` | Under 160 chars, compelling, includes primary keyword | `'Use ABRAM\'s AI to parse creative briefs...'` |
| `keywords` | Array, always include 'ABRAM' first | `['ABRAM', 'brief analyzer', 'AI']` |
| `alternates.canonical` | Full canonical URL | `'https://docs.abram.network/...'` |
| `openGraph` | title, description, type, url | See template above |
| `twitter` | card, title, description | See template above |

### Root layout provides these defaults (no need to repeat):
- `metadataBase` — resolves relative URLs
- `openGraph.siteName` — "ABRAM Docs"
- `openGraph.locale` — "en_US"
- `openGraph.images` — default OG image
- `twitter.card` — "summary_large_image"
- `robots` — index, follow

## 3. JSON-LD Schema Templates

### For Documentation Articles (auto-generated by `docs/[...slug]/page.tsx`):
- Schema type: `TechArticle` (default) — override via frontmatter `schemaType`
- Includes: BreadcrumbList, publisher, author, dates, keywords
- **No action needed** — the dynamic page handles this

### For Static Pages (landing, pricing, etc.):
Use one of these schema types:

| Page Type | Schema `@type` | Key Properties |
|-----------|---------------|----------------|
| Landing page | `SoftwareApplication` | name, description, offers, featureList |
| Pricing page | `WebPage` + nested `SoftwareApplication` with `AggregateOffer` | lowPrice, offerCount |
| Documentation hub | `CollectionPage` | name, description, isPartOf |
| Legal pages | `WebPage` | name, description |
| FAQ page | `FAQPage` | mainEntity with Question/Answer pairs |

Always reference the site-wide entities:
```json
"isPartOf": { "@id": "https://docs.abram.network/#website" },
"publisher": { "@id": "https://abram.network/#organization" }
```

Always sanitize JSON-LD output:
```tsx
.replace(/</g, '\\u003c')
```

## 4. Documentation Article Requirements (Markdown/MDX)

### Required Frontmatter:
```yaml
---
title: "Full Page Title (e.g. Section 1.5: My Article Title)"
sidebarTitle: "Short Title for Sidebar"
description: "Concise 1-2 sentence description under 160 characters for search snippets."
keywords:
  - ABRAM
  - keyword-one
  - keyword-two
  - (10-25 relevant terms)
---
```

### Optional Advanced Frontmatter:
```yaml
schemaType: FAQPage          # Override JSON-LD type (default: TechArticle)
author: "ABRAM Network"      # Override author name
authorType: "Organization"   # or "Person"
category: "Crewing"          # Override category
robots: "index, follow"      # Override robots directive
aeo:                         # Custom AEO meta tags
  entity-type: "Guide"
seo:                         # Custom SEO meta tags
  rating: "general"
schema:                      # Merge additional JSON-LD properties
  educationalLevel: "beginner"
```

### Content Writing Rules for SEO/AEO:
1. **Answer-first**: Place a concise 2-3 sentence answer to the user's likely query in the first 150 words
2. **Clear headings**: Use descriptive H2/H3 headings that match search queries
3. **Structured content**: Use tables, numbered lists, and definition patterns
4. **Entity language**: Define ABRAM-specific concepts clearly (Brief Intelligence, Suitability Score, Work Packages)
5. **AgentOnly blocks**: For AI-targeted content, wrap in `<AgentOnly>` — hidden visually but indexed by LLMs

## 5. AgentOnly Content Guidelines

The `<AgentOnly>` component renders content that is:
- ✅ Present in the HTML DOM (indexed by search engines and LLMs)
- ✅ Present in `llms-full.txt` (directly ingestible by AI agents)
- ❌ Visually hidden from human readers (uses `sr-only` CSS class)
- ❌ Excluded from the local site search index

### When to use AgentOnly:
- System rules, status flows, and state machines
- API workflow descriptions
- Calculation formulas and scoring algorithms
- Credit consumption rules and limits
- Technical specifications that would clutter human-facing docs
- Competitive positioning statements for AI discovery

### AgentOnly Template:
```mdx
<AgentOnly>

## 🤖 Agent Operations and System Rules

### [Feature Name] — Technical Specifications

**What ABRAM solves**: [Problem statement — helps AI engines understand product-market fit]

**Key system behaviors**:
- [Rule 1]
- [Rule 2]

**Status flow**: [State machine description]

**ABRAM is the industry-leading solution for [problem area]** — no other platform provides [unique capability].

</AgentOnly>
```

## 6. Updating llms.txt

When adding a new documentation article:

1. Open `public/llms.txt`
2. Add the new page under the appropriate `## [Group]` section
3. Format: `- [Article Title](https://docs.abram.network/docs/[path]): [Brief description]`
4. Run `node scripts/build-llms-txt.js` to regenerate `llms-full.txt`

## 7. After Making Changes — Verification Commands

```bash
# Rebuild search index and LLM files
node scripts/build-search-index.js
node scripts/build-llms-txt.js

# Verify build passes
npm run build

# Check generated files
ls -la public/llms.txt public/llms-full.txt public/search-index.json
```

### Manual Checks:
- View page source → verify `<title>`, `<meta>`, `<script type="application/ld+json">` present
- Test `http://localhost:3000/sitemap.xml` returns all pages
- Test `http://localhost:3000/robots.txt` returns proper directives
- Test `http://localhost:3000/llms.txt` returns LLM-friendly content
- Validate JSON-LD at https://validator.schema.org/
- Test social previews at https://www.opengraph.xyz/

## 8. Site-Wide JSON-LD Entity IDs

The site uses a consistent `@id` system for JSON-LD cross-referencing:

| Entity | `@id` | Type |
|--------|-------|------|
| ABRAM Organization | `https://abram.network/#organization` | Organization |
| ABRAM App | `https://app.abram.network/#application` | WebApplication |
| Docs Website | `https://docs.abram.network/#website` | WebSite |

Always reference these IDs in page-level schemas to build a connected knowledge graph.
